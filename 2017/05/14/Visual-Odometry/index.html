Visual Odometry

​ Today, let’s talk about a new topic named visual odometry. Visual Odometry is an important issue in computer vision , especially, in Automatic driving and SLAM. Because my research is focusing on pose estimation and augemented reality, i found there are a lot of same contents connect my research with visual odometry , so i think i should give a eye to this field. After a few weeks, i fall in love with visual odometry, it’s very interesting .Moreover, there are so many works in this field, i’m so happy that i am working on it !

What is visual odometry ?

​ Visual Odometry is the process of estimating pose of the moving object, like MAVs, Robot, Self-driving car etc. In this approach we have a camera (or many cameras) rigidly attached to a moving object, and our aim is to compute a 6-DOF trajectory using the video stream coming from this camera. When i have only one camera, this method is called Monocular Visual Odometry.

Why visual odometry ?

​ There are more than one ways to determine the trajectory of a moving robot, like wheel odometry, but visual odometry has more accurate trajectory estimates compared to wheel odometry. Therefore, VO can be used as a complement to other methods (GPS, IMU, Laser Odometry). But in monocular vo, we can only estimate the trajectory , unique only up to a scale factor, so you can say that you moved one unit in X, two units in Y. However, it can be addressed by Stereo Visual Odometry. REMEMBER, this post will still concentrate on Mono-VO.

Visual Odometry Pipeline

​ First of all, i’ll briefly describe a complete pipeline to establish a visual odometry here. Here is the pipeline :

  1. Capture images from video or cam : $ I^t $, $ I^{t+1} $;
  2. Undistort the above imges (optional);
  3. Use feature detection algorithm to detect features in $ I^t $;
  4. Track and match those features to $I^{t+1}$, if matched features drop below a certain threshold, tracking is failed, Reset !
  5. Motion estimation. Basically use five-point algorithm with RANSAC to compute the essiential matrix, then to estimate Pose ( R, t ) from the essential matrix
  6. Local optimization to refine pose.

Problem Formulation

​ According to Step 1, we take some consecutive images (or video) as our input to system. In case of a Monocular version, the set of images at discrete times k is denoted by
$$
I{0:n} = {I{0},…,I{n}} \tag{1}
$$
Then, two camera positions at adjacent time k-1 and k are related by a transform matrix :
$$
T{k} = \left[\begin{matrix}R{k,k-1} & t{k,k-1} \ 0 & 1\end{matrix}\right] \tag{2}
$$
Finally, we can get the set of camera poses $P{0:n} = {P{0},…,P_{n}}$ contains whole transformations of camera w.r.t. the initial coordinate frame at k=0, according to previous step 2.

In all, the main task in Visual Odometry is to compute the relative transformations $T{k}$ from images $I{k}$ and $I{k-1}$, then to recover full trajectory $P{0:n}$ for camera by $T_{k}$.

Motion Estimation (Pose)

​ This module is the key point and core computation step in a VO system. We will compute camera motion $T_{k}$ between the previous and current image according formula (2), then by concatenation of all these single movements, the full trajectory of the camera can be recovered. That is our aim ! Like Figure2 ,

​ Figure 2

​ To compute camera motion, there are two method to recover relative motion $T_{k}​$. Appearance-based method or Feature-based method . Appearance-based method always use the intensity information of all pixels in the two input images, therefore we get more direct information from orignal image, this can reveal the relationship between two images in low level. However, it causes a more expensive computation. Feature-based method only use salient and visual features extracted (or tracked) across the imges, we must detect and match (track) same features between consecutive frames, then to compute relative poses.

​ Feature-based motion estimation must robustly detect large number of feature points, some traditional computer vision methods can address this problem, like SIFT, SURF, ORB, FAST, these can be implemented by OpenCV. We want to compute relative transformation between two frames, it is neccessary to match features in both images. There are three cases: 2D-to-2D, 3D-to-3D, 3D-to-2D.
